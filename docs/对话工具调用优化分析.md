# AgentX 对话工具调用优化分析

> 分析日期：2026-04-25
> 分析范围：对话链路中的工具调用部分，包括 MCP 外部工具、内置工具、工具结果展示等

## 概述

AgentX 的工具调用链路覆盖了 MCP 协议的外部工具、注解扫描的内置工具、容器化部署、池化客户端、超时/限次/截断保护、敏感信息脱敏等。整体架构功能完备，但仍有多个可优化方向。本文档汇总性能、健壮性、Token 效率、错误处理与前端体验等维度的优化点，按优先级排序。

---

## 一、性能与并发

### 1. 三个独立工具线程池，无统一治理
- [AgentToolManager.java:39](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/AgentToolManager.java#L39) `MCP_INIT_EXECUTOR`（MCP 客户端预热）
- [ManagedMcpToolProvider.java:37](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/ManagedMcpToolProvider.java#L37) `TOOL_EXECUTION_EXECUTOR`（外部工具执行）
- [AbstractBuiltInToolProvider.java:35](../AgentX/src/main/java/org/xhy/application/conversation/service/message/builtin/AbstractBuiltInToolProvider.java#L35) 又一个 `TOOL_EXECUTION_EXECUTOR`（内置工具执行）

每个池都使用 `CallerRunsPolicy` 兜底——当队列满时回退到调用线程（即 SSE 流处理线程），互相影响、难以观测。

**建议**：统一通过 Spring `TaskExecutor` 配置，与现有 `memoryRecallTaskExecutor`、`sessionRenameTaskExecutor` 风格一致，便于在 yml 中调参。

---

### 2. PooledMcpClient 全方法 `synchronized(holder)` 串行化
[PooledMcpClient.java:48-52](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/PooledMcpClient.java#L48) 中 `executeTool` 也加了 holder 级别的锁。意味着同一 MCP server 即便允许并发请求，多个 agent 也会被强制排队。

`listTools` 已经有 TTL 缓存可以降低锁压力，但 `executeTool` 是热路径。

**建议**：只对 `cachedToolSpecifications` 等可变状态加锁；DefaultMcpClient 自身若线程安全，则去掉外层 `synchronized(holder)`。

---

### 3. `waitForToolReady` 同步阻塞轮询
[McpUrlProviderService.java:320-336](../AgentX/src/main/java/org/xhy/application/conversation/service/McpUrlProviderService.java#L320) 用 `Thread.sleep(200)` 轮询 SSE URL，最长阻塞 15s——而且就坐在 chat 请求线程上。

**建议**：
- 改成异步 `CompletableFuture`，完成后通过回调返回
- 或者把 `deployTool` + `waitForToolReady` 移到 session 创建/打开时预热，chat 入口只读结果

---

### 4. McpUrlProviderService 四套手写 TimedValue 缓存
- [globalToolCache, userContainerCache, deployedToolCache, reviewContainerCache](../AgentX/src/main/java/org/xhy/application/conversation/service/McpUrlProviderService.java#L30-L33)
- 都是 `ConcurrentHashMap<String, TimedValue<T>>`，无大小上限、无命中率统计、无 LRU
- 长时间运行后 `deployedToolCache` 可能膨胀（每个 `user × tool` 组合一个 entry）

**建议**：统一替换为 Caffeine（支持 `maxSize` / `expireAfterWrite` / `recordStats`），便于观测命中率与防止内存膨胀。

---

### 5. 缺少工具调用结果短期缓存
[ManagedMcpToolProvider.java:136-145](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/ManagedMcpToolProvider.java#L136) `wrapExecutor` 每次都直接 `delegateExecutor.execute`。当 LLM 在同一 turn 用相同参数重复调用同一工具（hallucination 重试场景常见）时，重复消耗。

**建议**：增加 `(toolName, argumentsHash) → result` 的短 TTL 缓存（如 60s），同 turn 内同参数直接返回上次结果。

---

## 二、限流与安全

### 6. 工具调用次数预算只有"全局总数"
[AgentToolManager.java:54](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/AgentToolManager.java#L54) `chat.tools.max-calls=6` 只控制总次数。缺少：
- **per-tool 上限**：防止 LLM 反复调用同一工具死循环
- **per-error 上限**：防止失败后无脑重试（虽然 `toolPolicyPrompt` 让模型自己控制，但属于软约束）

**建议**：在 `ManagedMcpToolProvider` 中维护 `Map<String, Integer> perToolCount` 与 `Set<String> failedToolFingerprints`，超限直接拒绝。

---

### 7. 结果截断按字符 substring，可能切割中文/JSON
[ManagedMcpToolProvider.java:171-179](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/ManagedMcpToolProvider.java#L171) 直接 `substring`，对结构化结果（JSON/Markdown 表格）会破坏语法，让模型把残破内容当真。

**建议**：
- 优先沿"完整 JSON 节点"或"完整段落"边界截断
- 或者对超长结果改为"返回摘要 + 引用 ID，让模型按需 deep-read"


## 三、Prompt / Token 效率

### 10. 工具调用历史无法进入 chat memory
[AbstractMessageHandler.java:1312-1318](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1312) `shouldInjectHistoricalMessage` 过滤掉所有非 `TEXT` 消息——包括 `TOOL_CALL`。

这意味着模型看不到上一轮自己用过哪些工具、得到什么结果，只能从 [buildRecentToolContextSection](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1658) 拿到 system prompt 里的"工具状态摘要"。

**后果**：跨 turn 的工具调用上下文丢失，模型可能反复调用同样工具拿同样结果。

**建议**：把历史工具调用按 LangChain4j 的 `ToolExecutionRequest` + `ToolExecutionResultMessage` 重建注入 chat memory，让模型完整看到自己上一轮的"思考-调用-结果"链路。

---

### 11. 工具目录提示与 function schema 重复
- [generateToolCatalogPrompt](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L169) 把 8 个工具的 `name/source/description/preset` 拼成 system prompt
- 同时 LangChain4j 已经把工具的完整 JSON Schema 作为 function calling 元数据传给 LLM

两份信息部分冗余、互相矛盾时让模型困惑。

**建议**：二选一：
- 主流走 function calling 时省掉 catalog（只保留"工具策略"）
- 或者关闭 LangChain4j 的工具元数据传递（用文本 catalog + ReAct 风格）

---

### 12. 内置工具定义缓存 key 包含 `updatedAt`
[AbstractBuiltInToolProvider.java:244-246](../AgentX/src/main/java/org/xhy/application/conversation/service/message/builtin/AbstractBuiltInToolProvider.java#L244) cacheKey 拼接了 `agent.getUpdatedAt()`，agent 任何字段更新（哪怕只是改了名字）都会击穿。

**建议**：key 仅基于真正影响工具定义的字段（`toolIds + knowledgeBaseIds + toolPresetParams hash`）。

---

## 四、错误处理 / 标识

### 13. 错误识别全靠"❌"前缀字符串
- [ToolPayloadUtils.isToolExecutionSuccessful](../AgentX/src/main/java/org/xhy/application/conversation/service/message/ToolPayloadUtils.java#L36)
- [classifyToolError](../AgentX/src/main/java/org/xhy/application/conversation/service/message/ToolPayloadUtils.java#L40) 用 `contains("超时" / "达到上限" / "不可用" / "中断")` 分类

底层 MCP 工具自己返回的 `Error: ...` / `failed: ...` 不会被识别为失败。

**建议**：引入 `ToolExecutionResult { success, errorCode, message, raw }` 结构化 envelope，"❌"前缀只是 UI 兜底。

---

### 14. `setSharedToolCallCounter` 的初始化时机脆弱
[ManagedMcpToolProvider.java:78](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/ManagedMcpToolProvider.java#L78) 默认计数器是 ManagedMcpToolProvider 自带的；只有在 [AbstractMessageHandler.prepareExternalToolProvider](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1042) 里才换成共享的。

如果将来有人新增"未走 prepareExternalToolProvider"的路径，内置工具与 MCP 工具会各自独立计数，绕过预算。

**建议**：改为构造时强制传入 counter，或在 `provideTools` 内自动接入共享 counter。

## 五、前端展示

### 15. 工具调用消息折叠后看不到摘要
[tool-call-details.tsx:84-91](../agentx-frontend-plus/components/tool-call-details.tsx#L84) 折叠态只显示"工具详情 / 展开查看"。

**建议**：折叠态展示 `工具名 · 状态徽标 · 耗时`（如 `search_web · 成功 · 320ms`），让用户不用展开就能扫读。

---

### 16. 同轮多工具调用拆成多条消息
- 后端 [AbstractMessageHandler.java:643-677](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L643) 每次 `onToolExecuted` 都新建一条 `MessageType.TOOL_CALL`
- 前端 chat-panel 把每条都渲染为独立气泡

复杂任务（10 个工具调用）时屏幕被工具卡片塞满。

**建议**：在前端把同一 turn 的连续 TOOL_CALL 折叠为一张卡片，点开可看每条详情。后端 `ToolPayloadUtils.buildMultiToolPayload` 已经支持多工具 payload，但流式期间没用到。

---

### 17. 工具执行进度无法可视化
只看到"执行工具：xxx"标题，但工具执行 5–10 秒期间无进度反馈。长时间工具（如代码搜索）用户体验差。

**建议**：在 `wrapExecutor` 中加耗时计时器，每秒推送一次"工具 xxx 已运行 Ns"事件到前端。

## 六、优先级建议

| 优先级 | 优化项 | 收益 | 风险 |
| --- | --- | --- | --- |
| **P0** | #10 工具调用历史进 chat memory | 减少重复调用、提升回答质量 | 中（需测试 LangChain4j tool message 兼容性） |
| **P0** | #5 工具结果短期缓存 + #6 per-tool 限次 | 阻止死循环 / 节省 token | 低 |
| **P0** | #15 / #16 前端工具折叠 + 摘要 | 长任务体验 | 低 |
| P1 | #2 PooledMcpClient 锁粒度 | 高并发吞吐 | 中（涉及并发正确性） |
| P1 | #3 waitForToolReady 异步化 | chat 首字节延迟 | 低 |
| P1 | #1 三个线程池统一治理 | 资源可控、可观测 | 中 |
| P1 | #11 catalog 与 function schema 二选一 | 节省 system token | 低 |
| P2 | #7 安全截断 / #8 脱敏扩展 | 数据正确性 / 合规 | 低 |
| P2 | #4 缓存换 Caffeine | 可观测、防膨胀 | 低 |
| P2 | #9 MCP 健康检查 | 长时运行稳定性 | 低 |
| P3 | #12 / #13 / #14 内部代码质量 | 可维护性 | 极低 |

## 八、推进建议

如果想推进，建议先聚焦 P0 三项之一：

1. **工具调用历史进 chat memory（#10）**：直接影响多轮工具任务的回答质量，最有潜在价值
2. **前端工具折叠（#15 + #16）**：改造范围小、体验提升立竿见影
3. **工具结果缓存 + per-tool 限次（#5 + #6）**：防止死循环、节省 token，对线上环境立刻产生收益

每项都需要：
- 单元测试覆盖（`MessageDomainServiceTest` / `ManagedMcpToolProviderTest`）
- 手工回归（普通对话 / 含工具调用对话 / RAG / Widget）
- 性能基线对比（如缓存命中率、平均工具调用次数）
