# AgentX 对话提示词与上下文优化分析

> 分析日期：2026-04-25
> 分析范围：对话链路中的"系统提示词装配"和"历史/记忆/RAG/附件上下文管理"两个维度，覆盖 PromptContextBuilder、PromptSectionPlan、buildHistoryMessage、ConversationAttachmentService、TokenEstimatorService 等

## 概述

AgentX 的对话提示词与上下文装配涉及多套机制协同：
- 静态 system 段：persona / tool_policy / tool_catalog / preset_tools / stable_memory
- 动态 system 段：summary / dynamic_memory / recent_tool_context / tool_availability_notice
- 历史消息：UserMessage / AiMessage 注入 chat memory
- 附件上下文：当前轮文本嵌入 + 历史轻量引用
- 预算控制：[applyPromptSectionBudget](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1414) 按优先级贪心丢弃

整体架构清晰，但存在多处可优化点。本文档按"装配架构 / 总长度控制 / 历史消息 / Token 估算 / 观测调试 / 提示词内容 / 附件上下文 / 多模型适配"8 个维度组织 35 个优化点，按 P0–P3 排序。

---

## 一、Prompt 装配架构

### 1. PromptContextBuilder 与预算策略顺序不一致
[PromptContextBuilder.java:52-61](../AgentX/src/main/java/org/xhy/domain/prompt/PromptContextBuilder.java#L52) 内部固定顺序：persona → capabilities → memory → rag → supplement → constraints。

但 [AbstractMessageHandler.java:1428-1438](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1428) `applyPromptSectionBudget` 又按另一个 `prioritySections` 列表做预算选择（persona / summary / tool_policy / availability_notice / dynamic_memory / stable_memory / preset / recent_tool / tool_catalog）。

两套排序观点存在重复或冲突时不易察觉。

**建议**：把"重要性优先级"统一表达在一处（PromptContextBuilder），删除 prioritySections 的隐式优先级；输出顺序与重要性顺序解耦（输出可保留 persona-first 的人类可读序，重要性由元数据携带）。

---

### 2. Section 是"全有或全无"丢弃，无截断回退
[AbstractMessageHandler.java:1500-1521](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1500) `selectPromptSection`：超预算直接整段跳过。

后果：
- 一个 800 token 的 `tool_catalog` 可能挤掉 100 token 的高价值 `dynamic_memory`
- 长 `preset_tool_prompt` 可能整段丢失，preset 工具就完全失效

**建议**：每个 section 声明"最低保留 token"和"完整 token"两档，超预算时先尝试压缩到最低档（如 catalog 从 8 个降到 3 个），再考虑丢弃。

---

### 3. Section 之间无依赖关系声明
- `tool_catalog` 依赖 `tool_policy`（前者描述工具列表，后者约束调用规则）
- `preset_tool_prompt` 与 `tool_availability_notice` 都谈工具，但分散
- 没有依赖图保证"丢了 policy 仍保留 catalog"不会发生

**建议**：在 `PromptSectionRef` 增加 `dependsOn`，预算选择时被依赖的 section 必须保留，否则整组丢弃。

---

### 4. system 段用连续多个 SystemMessage 注入
[AbstractMessageHandler.java:1286-1291](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1286) 把每个 section 都 `memory.add(new SystemMessage(...))`，最终发给 LLM 时是 N 条连续 SystemMessage。

- OpenAI 会自动合并多个 system message
- Anthropic 仅认第一个 system，其他被当作 user
- DeepSeek/通义千问行为各异

**建议**：在 `LLMServiceFactory` 层做归一化：对 Anthropic 等模型把多 system 合并成一条；对 OpenAI 保持原样，便于享受 system 段独立缓存。

---

### 5. PromptContextBuilder 没有 metadata，丢失类型信息
[PromptContextBuilder.java](../AgentX/src/main/java/org/xhy/domain/prompt/PromptContextBuilder.java) 只 `addSection(target, section)` 把字符串放进 list；下游 `applyPromptSectionBudget` 只能按"index + stable/dynamic"反推 section 类型，强耦合实现。

**建议**：定义 `PromptSection { type, content, priority, minTokens, maxTokens }` record，全链路传递。

---

## 二、总长度 / 预算控制

### 6. 预算公式硬编码 35%
[AbstractMessageHandler.java:1523-1539](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1523) `resolveSystemSectionBudgetTokens`：
- `budgetTokens = floor(availableTokens * 0.35)`
- `defaultBudget = 1200`, `minBudget = 320`, `maxBudget = 2600`

35% 是经验值；不同模型/Agent 类型应调整：
- 短上下文模型（4k）应给 system 更高比例（用户消息少）
- 长上下文模型（200k）系统提示占比可降低，多留给 history/RAG

**建议**：移到 [ChatContextProperties](../AgentX/src/main/java/org/xhy/application/conversation/config/ChatContextProperties.java) 配置；或按 `model.maxTokens` 分级（<8k=0.4, 8-32k=0.3, >32k=0.2）。

---

### 7. budget 不计 chat history 占用
[applyPromptSectionBudget](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1414) 只在 system sections 内部做预算。但 LLM 的总 token = system + history + current_user_message + reserve。

- 长 history 时 system + history 总和可能溢出
- 当前 history 是另一个机制（`MessageWindowChatMemory.maxMessages`）控制的，两者各管各的

**建议**：建立"总 prompt 预算"，自顶向下分配给 system / history / attachment / rag_context；任何一段超出从其他段挤压（结合 RAG 文档预算的 [RagAnswerPromptBuilder](../AgentX/src/main/java/org/xhy/application/conversation/service/message/rag/RagAnswerPromptBuilder.java#L130) 的思路推广到全局）。

---

### 8. 工具策略 prompt 每次重新生成
[generateToolPolicyPrompt(maxToolCallsPerTurn)](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L148-L167) 每次构造拼字符串，约 200 token、内容稳定。

- 没有缓存
- 没有标记为可 prompt-cache 的稳定段

**建议**：按 `maxToolCallsPerTurn` 缓存为常量；与 catalog 共同标记为 prompt cache 候选（参考 Anthropic prompt caching：`cache_control: {"type": "ephemeral"}`）。

---

### 9. 工具目录上限 8 个硬截断
[ConversationPromptTemplates.java:18](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L18) `MAX_TOOL_CATALOG_ITEMS=8`。

工具丰富的 Agent（>8 个工具）只能展示前 8 个，剩下 "已省略"——重要工具可能被截掉。

**建议**：
- 改为按 token 预算自适应（持续填充直到 budget 用完）
- 或按"最近调用过的优先 + 系统内置优先"加权选择
- 或按用户当前 query 与工具描述的 embedding 相似度排序

---

### 10. 工具目录描述长度上限 72 字符
[ConversationPromptTemplates.java:19](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L19) `MAX_TOOL_DESCRIPTION_LENGTH=72`。

72 字符（CJK 约 36 字）对复杂工具描述太短；工具用法关键约束被截。

**建议**：对工具描述做"概要 + 必要约束"两段式压缩，关键约束（如"仅在用户明确要求时调用"）必须保留。

## 三、历史消息装配

### 11. 仅 TEXT 历史进 chat memory（已在工具调用分析中提及）
[shouldInjectHistoricalMessage](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1312-L1318) 过滤掉 `TOOL_CALL` / `TOOL_NOTICE` / `RAG_*`。

模型看不到自己上一轮工具调用的入参/出参，跨 turn 推理能力受损（详见 [对话工具调用优化分析#10](对话工具调用优化分析.md)）。

---

### 12. 历史用户消息附件文本"硬合并"到 content
[buildHistoricalUserMessage](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1816-L1826) 把历史附件 `<attachments>...</attachments>` 文本块拼到 content 末尾。

- 长会话中前几轮的所有附件文本永远占据 token
- 没有"几轮之后丢弃"机制

**建议**：超过 N 轮（如 3）的历史附件文本只保留 "用户曾上传 X 张图片/文档"的一行简记。

---

### 13. 历史图片不重新作为多模态注入
[buildHistoricalImageReference](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1828-L1862) 历史图片仅作为文本 URL 引用。

后果：用户问"上面那张图里红色按钮在哪"时，模型已经看不到图，只能猜。

**建议**：
- 提供配置 `history.recentMultimodalTurns=2`（已存在 [#L39](../AgentX/src/main/resources/application.yml#L39)）让最近 N 轮图片仍以多模态形式注入
- 当前默认 `recent-multimodal-turns: 2` 配置生效，但代码中 `buildHistoricalUserMessage` 没看到使用——确认配置是否真的生效

---

### 14. 历史 AI 消息丢失工具调用元数据
[#L1305-L1308](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1305) 历史 ASSISTANT 都被注入为 `AiMessage(content)`。

LangChain4j 的 `AiMessage` 支持携带 `toolExecutionRequests`，当前只塞文本。模型不知道上一轮自己调过哪些工具。

**建议**：把历史 TOOL_CALL 消息按时间合并到对应 AiMessage 的 `toolExecutionRequests` 字段（需要数据库存储工具调用请求/响应的结构化关联，配合 [对话工具调用分析 #17](对话工具调用优化分析.md) 的改造）。

---

### 15. summary 与 chat history 的关系不清晰
当前流程：
- summary 单独作为 dynamic system 段注入
- summary 消息（role=SUMMARY）从 history 列表中过滤掉（`shouldInjectHistoricalMessage` 拒绝 summary）
- 但保存路径仍把 summary 放在 messageHistory 里

**建议**：明确 SummaryMessage 的角色定位——它不是 history 的一部分而是 system 段的一部分；从 messageHistory 中剥离单独管理。

---

### 16. 历史用户消息时间信息丢失
`buildHistoricalUserMessage` 仅保留 content，没有任何 "发自 1 小时前" / "本轮 5 分钟前" 的时间提示。

模型无法理解"昨天我说的方案"vs"刚才说的方案"语境差异。

**建议**：对超过 30 分钟的历史消息追加 `[相对时间: N 小时前]` 元注释（仅在 long-running session 启用）。

## 四、Token 估算

### 17. estimateByHeuristic 系数粗糙
[TokenEstimatorService.java:73](../AgentX/src/main/java/org/xhy/domain/token/service/TokenEstimatorService.java#L73) `estimated = cjk + latin/4 + digits/2 + others`。

实际：
- 中文一字常 1.5–2 token（GPT-4o）
- 日韩 1.5–3 token
- 该公式低估 CJK 成本约 50%

**建议**：
- 对 CJK 系数改为 1.5
- 对非 OpenAI provider 引入对应的本地 tokenizer（如 anthropic-tokenizer-jvm）

---

### 20. tokenEstimator 在工具消息上多次重复调用
[populateAssistantMessageTokenEstimate](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1148) 在每条 toolMessage 创建时调用一次。多工具调用场景累计 N 次。

工具消息不参与 LLM 计费（只是 metadata 落库），不需要精确估算。

**建议**：工具消息直接用启发式（不查 jtokkit/openai estimator），节省 CPU。

---

## 六、提示词内容质量

### 25. 工具策略 prompt 中文段落式
[generateToolPolicyPrompt](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L148-L167) 整段中文规则，没有分项编号。

模型对长段落的理解 < 短列表（OpenAI cookbook 建议）。

**建议**：改为编号列表：
```
<tool_policy>
1. 仅在需要外部信息时调用工具
2. 工具失败后允许重试 1 次，更多次直接基于现有信息作答
3. 本轮总预算：{N} 次
...
</tool_policy>
```

### 27. summary 8 行硬限制
[ConversationPromptTemplates.java:21](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L21) `MAX_SUMMARY_LINES=8`。

长任务（持续几十轮）的关键长期信息可能丢失。

**建议**：
- 分层 summary：近期 5 行 + 长期 3 行
- 或按 token 预算控制（如 300 token），让长 summary 不被截断

---

### 29. wrapStableMemorySection / wrapDynamicMemorySection 模板冗余
[ConversationPromptTemplates.java:92-104](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L92) 两个 section 各包一段中文说明（约 80 字 ×2 = 160 字 ≈ 80 token）。

加上 `<conversation_summary>` 的说明（[#L40-L42](../AgentX/src/main/java/org/xhy/domain/prompt/ConversationPromptTemplates.java#L40)）、RAG context 说明，总开销显著。

**建议**：把"使用规则"统一在一个 `<context_usage_policy>` section 中，各 context 段不再重复说明。

---

### 30. 用户消息与 system 段的角色边界模糊
[buildCurrentUserMessage](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1364-L1388) 把 attachments 文本块拼到用户消息末尾。

attachment 实际属于"用户提供的资料"，但 RAG 的 `<context>` 又是放在用户消息里（见 [RagAnswerPromptBuilder.buildUserPrompt](../AgentX/src/main/java/org/xhy/application/conversation/service/message/rag/RagAnswerPromptBuilder.java#L45-L50)）。两者混在 user role，模型不易区分"用户原话"vs"系统注入资料"。

**建议**：用 OpenAI tool message 或 system message 作为"参考资料"角色（多模型支持差异需要适配层）。

---

## 七、附件上下文

### 31. 文档解析在请求线程同步执行
[ConversationAttachmentService.buildDocumentSummary](../AgentX/src/main/java/org/xhy/application/conversation/service/ConversationAttachmentService.java#L196-L203) 调用 `fetchBytes`（5+8s 超时） + PDF/DOCX 解析（CPU 重）。

[AbstractMessageHandler.prepareCurrentTurnAttachments](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java#L1099-L1108) 在 chat 入口调用——直接阻塞 SSE 首字节延迟。

**建议**：
- 摘要异步生成（参考 memory prefetch 模式），用现有 attachment 存储缓存
- 上传时就预解析（OssUploadService 触发后台任务）

---

### 32. 文档摘要无缓存
同一附件 URL 每次对话都重新下载 + 解析。

**建议**：按 url 哈希缓存摘要（Caffeine maxSize=10000，TTL 24h）；上传完成后即写入缓存。

---

### 33. 图片 OCR 降级时同步阻塞
[buildImageFallbackSummaries](../AgentX/src/main/java/org/xhy/application/conversation/service/ConversationAttachmentService.java#L80-L112) 在主路径上调用 OCR 模型。

OCR 失败/超时影响主对话流程。

**建议**：图片到 chat 入口前就完成 OCR（上传时触发），失败标记为"OCR 不可用"但仍允许多模态注入。

---

### 34. attachment 文本无整体长度预算
[ConversationAttachmentService.buildCurrentTurnAttachmentText](../AgentX/src/main/java/org/xhy/application/conversation/service/ConversationAttachmentService.java#L126) 把所有附件 summary 拼接，无总长度上限。

5 个 PDF × 1200 字符 = 6000 字符 ≈ 1500–2000 token，可能挤掉 history。

**建议**：与 system budget 联动：attachment 总 token 上限 = available × 0.2。

## 十、推进建议

如果想推进，建议先聚焦 P0 四项：

1. **#35 SSRF 防护**（半天）：纯安全 fix，几行 URL 校验
