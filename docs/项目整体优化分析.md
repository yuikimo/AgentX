# AgentX 项目整体优化分析

## 概述

## 一、安全（最高优先级）

### 6. 拦截器排除路径硬编码两份
[WebMvcConfig.java:24-32](../AgentX/src/main/java/org/xhy/infrastructure/config/WebMvcConfig.java#L24) 和 [UserAuthInterceptor.java:27-29](../AgentX/src/main/java/org/xhy/infrastructure/auth/UserAuthInterceptor.java#L27) 维护两份相似但不一致的排除列表。

例如 `WebMvcConfig` 排除 `/sso/**` 整体，但 `UserAuthInterceptor.EXCLUDED_PATHS` 没有 sso；同样 `/widget/**` 在 WebMvcConfig 排除但 interceptor 字段没列。

**建议**：合并为单一 `AuthExcludePathProperties`（yml 配置），两边引用同一来源。

---

### 7. UserContext ThreadLocal 在异步任务中丢失
[UserContext.java](../AgentX/src/main/java/org/xhy/infrastructure/auth/UserContext.java) 是简单的 ThreadLocal。

`@Async` 方法 / `CompletableFuture.supplyAsync` 在新线程里执行时，`UserContext.getCurrentUserId()` 返回 null。代码中已经能看到 `MDC.getCopyOfContextMap()` + `runWithMdc` 模式，说明已经踩过坑，但 `UserContext` 本身没有同样的传递机制。

**建议**：
- 用 [TransmittableThreadLocal](https://github.com/alibaba/transmittable-thread-local) 替换，自动支持线程池传递
- 或者所有自定义 `TaskExecutor` 包一层 `TaskDecorator` 复制 UserContext + MDC

## 三、配置管理

---

### 13. 配置类散落多处不统一
- `chat.context.*` 用 [ChatContextProperties](../AgentX/src/main/java/org/xhy/application/conversation/config/ChatContextProperties.java)（@ConfigurationProperties）
- `chat.tools.*` 直接用 `@Value` 散布在多个类（[AgentToolManager.java#L54-L61](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/AgentToolManager.java#L54)、[ManagedMcpToolProvider.java](../AgentX/src/main/java/org/xhy/application/conversation/service/message/agent/ManagedMcpToolProvider.java)、[McpUrlProviderService.java](../AgentX/src/main/java/org/xhy/application/conversation/service/McpUrlProviderService.java) 等）
- `memory.*` 一部分用 [MemoryRecallProperties](../AgentX/src/main/java/org/xhy/domain/memory/config/MemoryRecallProperties.java)，一部分仍是 `@Value`（[MemoryDomainService.java#L78-L124](../AgentX/src/main/java/org/xhy/domain/memory/service/MemoryDomainService.java#L78)）

**建议**：统一为 `@ConfigurationProperties` 风格；`@Value` 仅用于一次性参数。便于 IDE 校验、IDEA Spring 插件提示、文档化。

## 五、API 治理

### 19. Controller 入参校验不统一
- 部分用 `@Validated` + `@NotBlank` / `@NotNull`（如 RAG 的 [RagSearchRequest.java](../AgentX/src/main/java/org/xhy/application/rag/dto/RagSearchRequest.java)）
- 部分用裸 `@RequestBody` 不校验
- 部分用 `@RequestParam @NotBlank` 直接在方法签名（[PortalAgentSessionController.java:69](../AgentX/src/main/java/org/xhy/interfaces/api/portal/agent/PortalAgentSessionController.java#L69)）

**建议**：CLAUDE.md 已经规定 `@RequestBody` 必须 `@Validated`，但没有自动检查。可以加一个 ArchUnit 规则强制。

---

### 20. 错误响应结构不利于前端处理
[Result.java](../AgentX/src/main/java/org/xhy/interfaces/api/common/Result.java) 只有 `code / message / data / timestamp`，缺少：
- `errorCode`（业务错误码，如 `MODEL_TIMEOUT` / `INSUFFICIENT_BALANCE`）
- `traceId`（排障）
- `details`（字段级错误，validation 时常用）

前端只能用 `code === 200` 这种简单判断，无法对不同业务错误做差异化提示（如余额不足跳转充值、模型超时显示重试按钮）。

**建议**：扩展为
```java
public class Result<T> {
    Integer code;
    String message;
    String errorCode;       // 业务错误码（成功时为 null）
    String traceId;
    Map<String, String> fieldErrors; // 字段错误
    T data;
    Long timestamp;
}
```

---

### 21. 接口路径风格不统一
- `/agents/sessions/{sessionId}/messages`（嵌套）
- `/admin/users`（管理员前缀）
- `/v1/...`（外部 API）
- `/widget/**`（公开）
- `/oauth/github/**`（OAuth 回调）
- 部分用驼峰 / 部分用 dash

**建议**：制定 API 风格指南（CLAUDE.md 已有部分），用 ArchUnit 检查 `@RequestMapping` 命名规则。

## 六、架构合规（DDD）

### 22. 应用服务过厚承担逻辑
CLAUDE.md 强调 `Application 层应当只协调 Domain`，但实际中：
- [ConversationAppService.java](../AgentX/src/main/java/org/xhy/application/conversation/service/ConversationAppService.java) 1069 行做了大量编排 + 业务逻辑
- [AbstractMessageHandler.java](../AgentX/src/main/java/org/xhy/application/conversation/service/message/AbstractMessageHandler.java) 2000+ 行混合 application + domain 逻辑

应用层下沉为薄编排层，业务规则归 Domain Service。

**建议**：用 ArchUnit 限制每个 AppService 类行数 ≤ 400 行；超过的强制重构。

---

### 23. Infrastructure 反向依赖隐患
CLAUDE.md 明确规定 Infrastructure 不能依赖 Application。值得抽样检查：
- `org.xhy.infrastructure.transport.MessageTransport` 接收 `org.xhy.application.conversation.dto.AgentChatResponse` ([MessageTransport.java#L3](../AgentX/src/main/java/org/xhy/infrastructure/transport/MessageTransport.java#L3))

`AgentChatResponse` 在 application 包，infrastructure 引用了它——**违反 DDD 分层规则**。

**建议**：
- `AgentChatResponse` 移到 `infrastructure.transport.dto` 或者 `domain.conversation.dto`
- 用 ArchUnit 增加规则：`infrastructure.* 不得 import application.*`

---

### 24. Repository 接口在 domain，但参数有 MyBatis-Plus 类型
domain Repository 应当 framework-agnostic，但实际继承 `MyBatisPlusExtRepository<Entity>`，签名带 `Wrapper<Entity>` 等 MyBatis 类型。

如果未来要切 JPA / R2DBC / 其他 ORM，整个 domain 层都要改。

**建议**：domain 暴露纯接口（如 `findByUserId(String userId, MemoryFilter filter)`），实现类放 infrastructure 用 MyBatis-Plus 实现。

---

### 25. interfaces 层 DTO 与 application 层 DTO 重叠
[interfaces/dto/conversation/request/ConversationRequest.java](../AgentX/src/main/java/org/xhy/interfaces/dto/conversation/request/ConversationRequest.java) vs [application/conversation/dto/ChatRequest.java](../AgentX/src/main/java/org/xhy/application/conversation/dto/ChatRequest.java) 角色易混。

**建议**：明确：
- `interfaces.dto.*Request` = HTTP wire format（含 `@Validated`）
- `application.dto.*Command` = use case 入参（领域意图）
- Controller 把 Request 转 Command 后调 AppService

## 七、部署 / 运维

### 26. docker-compose 凭据明文
[deploy/docker-compose.yml](../deploy/docker-compose.yml) 没读取，但根据 application.yml 默认值推断很可能也含明文凭据。

**建议**：用 docker secrets / k8s secrets，不在 yml 内 inline。

---

### 27. 日志写本地文件
[application.yml#L116-L117](../AgentX/src/main/resources/application.yml#L116) `logging.file.name: logs/agent-x.log`。

容器化部署时本地日志会随容器删除而消失；多副本时无法聚合。

**建议**：仅输出 stdout（k8s/docker 自动收集），由集群层日志方案（fluent-bit + Loki）处理持久化。

---

### 28. 没有看到优雅停机配置
SSE 连接长达 5 分钟，`spring-boot-starter-actuator` 的 graceful shutdown 没启用，Pod 重启可能切断进行中对话。

**建议**：
```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 60s
```

---

### 29. RabbitMQ 关键参数未保护
[application.yml#L66](../AgentX/src/main/resources/application.yml#L66) `concurrency: 19, max-concurrency: 19`，硬编码 19 看似随意（不是 1/2/4/8/16/20 这种常见值）。

**建议**：参数化为环境变量 + 文档说明为何 19。

## 八、前端工程化

### 30. 前端无 ESLint / Prettier 强制
没有看到 `.eslintrc` / `.prettierrc`（未直接读取，但若有 README 应提到）。

代码风格不一致（前面 chat-panel.tsx 中混用单/双引号、缩进偶尔不齐）。

**建议**：引入 `eslint-config-next` + `prettier`，配 husky + lint-staged 在 commit 时强制。

---

### 31. 前端 API 服务文件巨大
[lib/api-services.ts](../agentx-frontend-plus/lib/api-services.ts)（前面看过 Glob）以及散落的 *-service.ts。

未做模块边界划分，类型定义有重复（如 `Message` 类型多处定义，前面 chat-panel.tsx 与 types/conversation.ts 不一致）。

**建议**：
- 用 `springdoc-openapi` 生成 API 客户端代码（自动同步）
- 或者至少每个 controller 对应一个 service.ts，类型集中在 types/

---

### 32. 前端缺少错误边界与 Suspense
没有 `<ErrorBoundary>` / `<Suspense>` 包裹路由。任何组件抛错会让整个页面白屏。

**建议**：app/(main)/layout.tsx 加全局 error.tsx；关键异步组件用 React Suspense + fallback。

---

### 33. 前端用 fetch 而非统一 client
[lib/stream-service.ts](../agentx-frontend-plus/lib/stream-service.ts) 用裸 `fetch`，[lib/http-client](../agentx-frontend-plus/lib/http-client) 是另一套。

两套并存增加维护成本。

**建议**：合并为一套（保留 SSE 路径，但通用 fetch 拦截器统一处理 401/网络错/重试）。

---

## 九、文档与协作

### 34. CLAUDE.md / AGENTS.md 内容详尽但执行力弱
[CLAUDE.md](../CLAUDE.md) 包含详细的代码规范、API 标准、命名约定，但落地缺乏强制：
- 未启用 ArchUnit
- Spotless 配置存在但 CI 未必跑
- 没有 commit message 校验

**建议**：
- pre-commit hook 跑 `mvn spotless:check`
- CI 跑 ArchUnit 检查 DDD 分层
- 引入 commitlint / commitizen

---

### 35. 缺少 ADR（Architecture Decision Records）
为什么选 langchain4j、为什么自建容器管理而非 k8s job、为什么用 pgvector 而非 Milvus……都没有书面记录。新成员加入时只能口头传授。

**建议**：在 docs/adr/ 下维护 ADR，每个重大架构决策一份（参考 [adr.github.io](https://adr.github.io/)）。

## 十二、推进建议

如果想推进，**强烈建议先做 P0 五项**（凭据 + JWT + 限流 + 错误脱敏 + 烟雾测试），它们都是上线前必须解决的安全/质量门槛：

1. **#1 + #2 凭据清理**（半天）：所有 secret 改为环境变量必传 + 清理 git 历史 + 轮换所有第三方 key
2. **#5 通用限流**（1–2 天）：定义 `@RateLimited` 注解 + Redis 实现 + 在 chat/login/upload 关键入口加上
3. **#8 错误脱敏**（半天）：GlobalExceptionHandler 区分 dev/prod，prod 不返回 e.getMessage 给客户端
4. **#9 烟雾测试**（1–2 天）：用 Testcontainers + 1 个 e2e 测试覆盖登录→对话→工具调用→注销

完成 P0 后，建议把可观测性（#14 / #15）作为下一阶段重点，否则后续优化效果难以验证。
